  # **`gcTime`**과 **`staleTime`**의 차이점에 대해 정리해보세요 🍠
    <aside>
    🍠
    
    gcTime과 staleTime은 Tanstack Query에 있어서 정말 중요하게 알고 넘어가야 하는 부분이에요.
    
    **위에서 제가 설명해드렸지만 gcTime**과 **staleTime**의 개념을 다시 정리해주시고, 두 값을 어떤 식으로 설정하면 캐싱 전략에 유리한지 설명해주세요!
    
    - UMC 중앙 웹 파트장 매튜 / 김용민 - 
    
    </aside>
    
    - `gcTime`은 무엇인가요? 🍠
        
        gcTime(Garbage Collection Time)은 **쿼리가 더 이상 사용되지 않을 때, 캐시 데이터를 메모리에서 얼마나 오래 유지할지** 결정하는 시간이다.
        
        기본값은 **5분(300,000ms)**이다.
        
        ```tsx
        useQuery({
          queryKey: ['todos'],
          queryFn: fetchTodos,
          gcTime: 5 * 60 * 1000, // 언마운트 후 5분 뒤 캐시 삭제
        });
        ```
        
        컴포넌트가 언마운트된 후 gcTime이 지나기 전에 다시 마운트되면, 기존 캐시를 그대로 재활용할 수 있다.
        
    - `staleTime`은 무엇인가요? 🍠
        
        staleTime은 **캐시된 데이터가 "신선한(fresh)" 상태로 유지되는 시간**이다.
        
        기본값은 **0ms**로, 데이터를 가져오는 즉시 stale 상태가 된다.
        
        ```tsx
        useQuery({
          queryKey: ['todos'],
          queryFn: fetchTodos,
          staleTime: 30_000, // 30초 동안은 fresh → 리페치 안 함
        });
        ```
        
        | 상태 | 의미 |
        | --- | --- |
        | **fresh** | 아직 신선함 → 리페치 트리거가 와도 네트워크 요청 안 함 |
        | **stale** | 신선도 만료 → 포커스 복귀, 네트워크 재연결 등 트리거 시 리페치 발생 |
    - 두 값을 어떤 식으로 설정하여야 `캐싱 전략에 유리`한가요? 🍠
        
        ### 핵심 원칙
        
        > **gcTime은 항상 staleTime보다 길게 설정한다.**
        > 
        
        staleTime이 gcTime보다 길면, 데이터가 아직 신선한데 캐시 자체가 메모리에서 삭제되는 모순이 발생하기 때문이다.
        
        ---
        
        ### 데이터 성격에 따른 전략
        
        **① 자주 변하지 않는 정적 데이터** (카테고리, 설정값 등)
        
        ```tsx
        useQuery({
          queryKey: ['categories'],
          queryFn: fetchCategories,
          staleTime: 10 * 60 * 1000,  // 10분 동안 fresh
          gcTime: 30 * 60 * 1000,     // 30분 동안 캐시 유지
        });
        ```
        
        staleTime을 길게 줘서 불필요한 네트워크 요청을 줄이고, gcTime도 넉넉히 설정해 캐시 재활용률을 높인다.
        
        ---
        
        **② 실시간성이 중요한 데이터** (알림, 재고 수량 등)
        
        ```tsx
        useQuery({
          queryKey: ['notifications'],
          queryFn: fetchNotifications,
          staleTime: 0,                // 즉시 stale → 항상 최신 데이터
          gcTime: 1 * 60 * 1000,      // 1분 후 캐시 정리
        });
        ```
        
        staleTime을 0으로 유지해 항상 최신 데이터를 가져오고, gcTime은 짧게 설정해 메모리 낭비를 방지한다.

    **HINT: 다음 주차 스터디 내용**
    
    </aside>
    
    - 답변
        
        <aside>
        ❓
        
        **나의 답변:**
        
        항상 캐시만 보여주면 방금 추가/수정/삭제한 데이터가 화면에 반영되지 않는 문제가 생긴다. 이를 해결하는 핵심 패턴이 바로 다음 주차에서 배울 **`invalidateQueries`**다. 뮤테이션(데이터 변경) 성공 시 해당 `queryKey`의 캐시를 강제로 무효화(stale 처리)하면, React Query가 즉시 백그라운드에서 최신 데이터를 다시 가져온다. 즉, 캐시의 빠름과 최신성을 동시에 확보하는 전략이다.

- **미션 3 - QueryClient 설정 읽기**
    <aside>
    ❓
    
    아래는 React Query를 사용할 때 가장 먼저 설정하는 **QueryClient** 코드예요. 코드를 읽고 질문에 답해보세요.
    
    ```tsx
    const queryClient = new QueryClient({
      defaultOptions: {
        queries: {
          staleTime: 1000 * 60 * 5, // 5분
          gcTime: 1000 * 60 * 6,    // 6분
        },
        mutations: {
          retry: false,
        },
      },
    });
    ```
    
    **Q1. `staleTime: 5분`은 무슨 뜻인가요?**
    
    **Q2. `gcTime: 6분`은 `staleTime`과 뭐가 다른가요?
    
    Q3. `mutations: { retry: false }`는 왜 설정했을까요?**
    
    </aside>
    
    - 답변
        
        <aside>
        ❓
        
        답변
        
        **Q1 답변.** 
        
        `staleTime: 5분`은 데이터를 서버에서 받아온 후 **5분 동안은 fresh(신선한) 상태로 간주**한다는 뜻이다. 5분 안에 같은 쿼리가 다시 실행되면 네트워크 요청 없이 캐시를 그대로 사용한다.
        ****
        **Q2 답변.**
        
        `gcTime: 6분`은 해당 쿼리를 사용하는 컴포넌트가 언마운트된 후 **6분이 지나면 캐시를 메모리에서 삭제**한다는 뜻이다. staleTime은 "데이터의 신선도(언제까지 재요청 안 해도 되는지)"를 제어하고, gcTime은 "캐시 자체의 수명(메모리에서 언제 지울지)"을 제어한다. gcTime이 staleTime보다 1분 더 길게 설정된 것은, 데이터가 stale 상태가 되더라도 캐시가 살아있어야 백그라운드 리페치가 가능하기 때문이다.
        
        **Q3 답변.**
        
        `mutations: { retry: false }`는 뮤테이션(POST, PUT, DELETE 등 데이터 변경 요청)이 실패했을 때 **자동으로 재시도하지 않겠다**는 설정이다. 조회와 달리 변경 요청은 같은 요청을 두 번 보내면 중복 생성이나 이중 결제 같은 심각한 부작용이 생길 수 있기 때문에, 실패 시 재시도를 막고 유저에게 명확한 에러를 전달하는 것이 안전하다.
        
        </aside>

- **미션 4 - Query Key는 왜 배열일까?**
    <aside>
    ❓
    
    React Query는 데이터를 키(key)로 구분해서 캐시에 저장해요. Aido의 할 일 기능을 예시로 살펴볼게요.
    
    “아이두” 앱에서는 할 일 데이터를 여러 화면에서 보여줘요:
    
    - **할 일 목록** — 오늘 날짜의 할 일 리스트
    - **캘린더 달성률** — 이번 달 날짜별 완료 표시
    
    이 두 화면은 서로 다른 API를 호출하지만, 둘 다 "할 일" 데이터예요. 이걸 `Query Key`로 이렇게 표현합니다:
    
    ```tsx
    // "아이두"의 할 일 Query Key
    const TODO_KEYS = {
      all:        ['todo'],                                       // 할 일 전체
      list:       (date: string) => ['todo', 'list', date],       // 특정 날짜의 할 일 목록
      completion: (date: string) => ['todo', 'completion', date], // 특정 날짜의 달성률
    };
    ```
    
    **Q1. 할 일을 하나 추가한 뒤 아래 코드를 실행하면 어떤 일이 일어날까요?**
    
    아래 코드는 다음 주차 워크북에서 배울거에요, 처음 보는 내용일 수 있지만 한번 **`invalidateQueries`**가 뭔지 검색을 통해 찾아보고 답변해보세요.
    
    ```tsx
    queryClient.invalidateQueries({ queryKey: ['todo'] });
    ```
    
    </aside>
    
    - 답변
        
        <aside>
        ❓
        
        답변
        
        Tip. 하나의 화면을 보여주기 위해서 여러 데이터를 할 일 화면에서 호출하는데 어떤식으로 동작할 것 같은지 생각해보고 답변해주세요.
        
        **Q1 답변.** 
        
        `queryClient.invalidateQueries({ queryKey: ['todo'] })`를 실행하면 `['todo']`로 시작하는 **모든 쿼리 캐시가 한 번에 stale 처리**된다. 즉, `['todo', 'list', date]`와 `['todo', 'completion', date]` 모두 무효화되어 각 화면이 다음 방문 시 서버에서 최신 데이터를 다시 가져온다.
        
        할 일 화면은 여러 API를 동시에 호출한다(목록, 달성률 등). 할 일을 하나 추가하면 목록과 달성률이 모두 바뀌는데, `['todo']` 하나만 무효화해도 관련된 모든 쿼리가 갱신된다. 쿼리 키를 계층 구조로 설계한 덕분에 세밀한 무효화와 일괄 무효화를 상황에 맞게 선택할 수 있다.
        
        </aside>

- **미션 5 - enabled** **속성의 중요성**
    <aside>
    ❓
    
    `useQuery`에는 **"지금은 요청하지 마"**라고 말할 수 있는 `enabled` 옵션이 있어요. 아래 예제를 보고 질문에 답해보세요.
    
    ```tsx
    function UserProfile() {
      const [userId, setUserId] = useState(null);
    
      const { data: profile } = useQuery({
        queryKey: ['user', userId],
        queryFn: () => fetch(`/api/users/${userId}`).then(res => res.json()),
        enabled: userId !== null,  // ← userId가 있을 때만 요청!
      });
    
      return (
        <>
          <button onClick={() => setUserId('abc123')}>프로필 보기</button>
          {profile && <p>{profile.name}</p>}
        </>
      );
    }
    ```
    
    **Q1.** `enabled: userId !== null`이 없으면 어떤 문제가 생길까요?
    
    **Q2. 실제로 지금 우리가 보고있는 “아이두” 서비스에서는 이런게 어디에 쓰일 수 있을 것 이라고 생각하시나요? 모든 화면들을 보면서 최대한 많이 찾아주세요.**
    
    </aside>
    
    - 답변
        
        <aside>
        ❓
        
        답변
        
        **Q1 답변.** 
        
        `enabled` 옵션이 없으면 컴포넌트가 마운트되는 순간 `userId`가 `null`인 채로 `/api/users/null`에 요청을 보낸다. 서버에서 잘못된 요청으로 에러가 반환되고, 불필요한 네트워크 요청과 에러 처리 로직이 발생한다.
        ****
        **Q2 답변.**
        
        - **로그인 여부 확인 후 데이터 요청** — 인증 토큰이 없는 상태에서 API를 호출하면 401 에러가 발생한다. 토큰이 존재할 때만 요청하도록 `enabled: !!accessToken`으로 막을 수 있다.
        - **위치 정보 기반 날씨 요청** — 위도/경도가 아직 없는 상태에서 기상청 API를 호출하면 의미 없는 요청이 된다. `enabled: !!latitude && !!longitude`로 위치가 확보된 후에만 요청한다.
        - **친구 프로필 조회** — 특정 친구를 선택한 후에만 해당 친구의 데이터를 요청해야 하므로 `enabled: !!selectedFriendId`로 제어한다.
        - **캘린더 날짜 선택 후 할 일 조회** — 날짜가 선택되지 않은 상태에서 요청하면 의미 없는 결과가 온다. `enabled: !!selectedDate`로 제어한다.
        </aside>

- **종합 과제 - 날씨 기능 분석하기**
    <aside>
    ❓
    
    "아이두" 에는 할 일을 잘 설정하기 위에서는 외부의 요인이 정말 크게 작용한다고 생각하고, 이런 외부 요인을 하나의 앱에서 해결하기 위해 여러가지 기능을 붙이고 있어요 그 중 첫번째가 날씨였고, 앱 내에는 기상청 API를 활용한 **날씨 기능**이 있어요. 지금까지 배운 패턴이 여기에 어떻게 적용되었는지 분석해보세요. (혹시나 날씨 기능이 보이지 않는다면 앱을 업데이트 해주세요)

    Q1. 
    
    날씨 화면에 처음 방문하면, 실제로 날씨 정보를 못 받고 있어요. 기상청은 **위도/경도**를 기반으로 날씨를 제공하는데, 아직 위치 정보가 없는 상태거든요. 그런데 위치 정보가 없는 상태에서 매번 이 화면을 방문할 때마다 서버에 날씨 요청을 보낸다면, **불필요한 요청이 계속 발생**해서 비용적으로 손해예요. 이 경우에 클라이언트에서 할 수 있는 방법은 어떤 게 있을까요?

    - 날씨에도 똑같이 **5분**을 적용하면 어떤 문제가 있을까요?
    - 그렇다고 기상청 갱신 주기와 동일하게 **3시간**으로 설정한다면 어떤 문제가 있을까요?
    - 어느 정도가 적절할까요?
    
    Q3.
    
    위치를 서울에서 부산으로 변경했어요. 이때 **서울의 날씨 캐시**가 아직 남아있다면, 부산 날씨를 보여줘야 하는데 서울 날씨가 보이는 문제가 생길 수 있어요. 이걸 어떻게 해결할 수 있을까요? 앞서 배운 개념 중에 사용할 수 있는 게 있을까요?
    
    </aside>
    
    - 답변
        
        <aside>
        ❓
        
        답변
        
        **Q1 답변:**
        
        `enabled` 옵션을 활용해 위치 정보가 확보된 후에만 API를 호출하도록 제어한다. `enabled: !!latitude && !!longitude`로 설정하면 위도/경도가 없는 상태에서는 쿼리 자체가 실행되지 않아 불필요한 요청을 완전히 차단할 수 있다.
        ****
        **Q2 답변:** 
        
        - **5분으로 설정하면?** 기상청은 3시간마다 데이터를 갱신하는데, 5분마다 똑같은 데이터를 계속 요청하게 된다. 사용자 입장에서는 차이도 없는 데이터를 위해 불필요한 API 호출이 반복되고, 기상청 API 호출 횟수 제한에 걸릴 수도 있다.
        - **3시간으로 설정하면?** 기상청 갱신 주기와 동일해 이론적으로는 맞지만, 실제로 정각에 갱신되지 않거나 네트워크 지연이 있을 때 최신 데이터를 놓칠 수 있다. 또한 앱을 오래 켜두었을 때 데이터가 너무 오래 묵을 수 있다.
        - **적절한 값은?** 30분~1시간 정도가 합리적이다. 기상청 갱신 주기보다 짧아 비교적 최신 데이터를 제공하면서도, 5분보다 훨씬 길어 불필요한 API 호출을 크게 줄일 수 있다.
        
        **Q3 답변:**
        
        `queryKey`에 위치 정보(위도/경도 또는 지역명)를 포함시키면 해결된다.
        
        ```tsx
        useQuery({
          queryKey: ['weather', latitude, longitude],
          queryFn: () => fetchWeather(latitude, longitude),
          staleTime: 30 * 60 * 1000,
        });
        ```
        
        서울과 부산은 좌표가 다르므로 서로 다른 캐시 키를 가진다. 부산으로 위치가 변경되면 `['weather', 부산위도, 부산경도]` 키로 새 요청이 발생하고, 서울 캐시는 별도로 유지되어 충돌하지 않는다.
