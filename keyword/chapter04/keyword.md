  # 📚 블로그 읽고 세션과 토큰 정리해보기

  ***

  [[🔐인증] 2. 세션 vs 토큰: 사용자의 상태를 어디에 저장할까](https://velog.io/@cucumtomatobean/dtd9x1yh)
  - 세션 방식과 토큰 방식의 가장 큰 차이는 무엇인가요? 🍠
    가장 핵심적인 차이는 **"상태 정보를 어디에 저장하는가"**입니다.
    - **세션 (서버 중심):** 사용자의 로그인 상태를 **서버**(메모리, DB 등)에 저장합니다. 클라이언트에게는 그 정보가 담긴 '상태'가 아니라, 그 상태를 찾을 수 있는 '열쇠(세션 ID)'만 줍니다.
    - **토큰 (클라이언트 중심):** 사용자의 로그인 상태를 **토큰 자체**에 담아서 **클라이언트**가 들고 있게 합니다. 서버는 상태를 저장하지 않고, 클라이언트가 가져온 토큰이 유효한지만 검사합니다.
  - 세션을 분산 환경에서 관리할 때 사용할 수 있는 세 가지 전략(Sticky Session, Session Replication, Centralized Session Store)의 특징을 정리해주세요. 🍠
    서버가 여러 대일 때 세션 정보가 공유되지 않는 문제를 해결하는 방법들입니다.
    | **전략**                | **특징**                                                            | **장점**                                      | **단점**                                                              |
    | ----------------------- | ------------------------------------------------------------------- | --------------------------------------------- | --------------------------------------------------------------------- |
    | **Sticky Session**      | 특정 사용자의 요청을 처음 로그인한 서버로만 계속 보내는 방식        | 구현이 매우 간단함                            | 해당 서버 장애 시 세션 증발, 특정 서버에 부하 집중 가능성             |
    | **Session Replication** | 모든 서버가 서로의 세션 정보를 복사해서 똑같이 나눠 가짐            | 어떤 서버로 가도 세션 존재                    | 서버가 많아질수록 복사 과정에서 네트워크 부하와 메모리 낭비 심화      |
    | **Centralized Store**   | 별도의 중앙 창고(Redis 등)를 두고 모든 서버가 거기서 세션을 꺼내 씀 | **현재 표준.** 서버 확장과 관리가 매우 유연함 | 별도의 저장소(Redis 등) 관리 비용 발생, 저장소 장애 시 전체 인증 마비 |
  - JWT(Json Web Token)의 장점과 단점을 각각 설명해주세요. 🍠
    - **장점 (확장성):** 서버가 사용자 상태를 저장하지 않으므로(Stateless), 서버가 수백 대로 늘어나도 각 서버가 비밀 키만 있으면 독자적으로 인증을 처리할 수 있습니다. MSA 환경에 최적화되어 있습니다.
    - **단점 (제어권 부족 & 크기):** 한 번 발급된 토큰은 서버가 회수할 방법이 거의 없습니다. 또한, 정보가 토큰에 포함되므로 세션 ID보다 데이터 크기가 커서 네트워크 부하가 생길 수 있습니다.
  - JWT의 즉시 무효화 문제가 생기는 이유와 이를 해결하기 위한 방법은 무엇인가요? 🍠
    - **이유:** 서버가 세션을 들고 있지 않기 때문입니다. 세션은 서버에서 지우면 끝이지만, JWT는 클라이언트가 들고 있는 "유효 기간이 남은 티켓"과 같아서 서버가 "그 티켓 이제 쓰지 마!"라고 강제로 뺏어올 방법이 없습니다.
    - **해결 방법:** \* **블랙리스트:** 로그아웃한 토큰 ID를 DB나 Redis에 등록해두고, 요청이 올 때마다 블랙리스트에 있는지 확인합니다.
      - **짧은 유효기간:** Access Token의 수명을 아주 짧게(예: 30분) 잡아 피해 시간을 최소화하고, Refresh Token으로 갱신하게 합니다.
  - 세션과 토큰을 결합한 하이브리드 방식(JWT + 저장소 메타데이터)의 동작 원리를 간단히 설명해주세요. 🍠
    세션의 **통제권**과 토큰의 **확장성**을 합친 방식입니다.
    1. **발급:** JWT 안에 `jti`(토큰 고유 ID)라는 클레임을 넣어서 발급합니다.
    2. **저장:** 서버(Redis 등)에는 토큰의 전체 정보가 아닌, 이 `jti`와 상태값(유효 여부)만 가볍게 저장합니다.
    3. **검증:** 요청이 오면 1차로 JWT 서명을 확인(토큰 방식)하고, 2차로 해당 `jti`가 서버 블랙리스트에 있는지 확인(세션 방식의 장점)합니다.
    4. **결과:** 평상시에는 토큰처럼 빠르게 검증하다가, 필요할 때(로그아웃, 강제 종료 등)만 특정 토큰 ID를 차단하여 즉시 무효화가 가능해집니다.
    **[JWT + Redis 하이브리드 방식의 핵심 이유]**
    JWT의 장점은 "서버가 정보를 안 가져도 된다"는 것이지만, 단점은 "사용자를 강제로 로그아웃시킬 권한이 없다"는 것입니다. 그래서 **Redis에 '블랙리스트'나 '허용된 토큰 리스트'를 저장**하여, 토큰은 토큰대로 쓰되 **서버가 통제권(Kill Switch)을 갖기 위해** 이 방식을 씁니다.

  [[🔐인증] 1. HTTP는 무상태인데 로그인 상태는 어떻게 유지할까](https://velog.io/@cucumtomatobean/%EC%9D%B8%EC%A6%9D-1.-HTTP%EB%8A%94-%EB%AC%B4%EC%83%81%ED%83%9C%EC%9D%B8%EB%8D%B0-%EB%A1%9C%EA%B7%B8%EC%9D%B8-%EC%83%81%ED%83%9C%EB%8A%94-%EC%96%B4%EB%96%BB%EA%B2%8C-%EC%9C%A0%EC%A7%80%ED%95%A0%EA%B9%8C)
  - HTTP는 왜 무상태(Stateless)로 설계되었나요? 🍠
    웹의 초창기 목적은 **'단순 문서 전달'**이었기 때문입니다.
    - **설계 이유:** 전 세계의 수많은 클라이언트가 접속할 때, 서버가 각 연결의 '기억'을 붙잡고 있으면 서버의 메모리가 금방 바닥나게 됩니다. 서버가 클라이언트를 기억하지 않게 함으로써, 요청이 들어올 때마다 **빠르게 응답하고 바로 잊어버리는** 효율적인 구조를 선택한 것입니다.
  - HTTP의 무상태성이 주는 장점과 단점을 각각 정리해주세요. 🍠
    | **구분** | **내용**                                                                      | **비고**                          |
    | -------- | ----------------------------------------------------------------------------- | --------------------------------- |
    | **장점** | **확장성(Scalability):** 어떤 서버가 요청을 받아도 상관없어 서버 증설이 쉬움. | 로드밸런싱에 유리                 |
    |          | **단순성:** 서버가 상태를 관리할 필요가 없어 구조가 단순함.                   |                                   |
    | **단점** | **연속성 부재:** 로그인 상태 유지, 장바구니 기능을 기본적으로 구현 불가.      | 이를 해결하기 위해 쿠키/세션 도입 |
    |          | **데이터 중복:** 상태를 증명하기 위해 매 요청마다 인증 정보를 계속 보내야 함. | 네트워크 부하 발생                |
  - 쿠키의 Domain 디렉티브에 대해 정리해주세요. 🍠
    쿠키가 어느 도메인까지 전송될지 결정합니다.
    - **설정 안 할 때:** 쿠키를 생성한 **현재 도메인**에서만 전송됩니다.
    - **설정할 때:** `Domain=yolog.co.kr`처럼 설정하면, 해당 도메인뿐만 아니라 **서브도메인**(`login.yolog.co.kr`, `blog.yolog.co.kr`)에서도 쿠키를 함께 사용할 수 있습니다. (SSO 구현 시 중요!)
  - 쿠키의 Path 디렉티브에 대해 정리해주세요. 🍠
    도메인 내부에서 특정 경로로 요청할 때만 쿠키를 보내도록 제한합니다.
    - 예: `Path=/private` 설정 시, `/private/settings` 요청에는 쿠키가 가지만, `/public/home` 요청에는 쿠키가 가지 않습니다. 불필요한 쿠키 전송을 막는 용도입니다.
  - 세션 쿠키와, 영속 쿠키의 차이점을 정리해주세요. 🍠
    - **세션 쿠키 (Session Cookie):** 만료 시간(`Expires`, `Max-Age`)을 설정하지 않은 쿠키. **브라우저를 닫으면 사라집니다.**
    - **영속 쿠키 (Permanent Cookie):** 만료 시간을 명시적으로 설정한 쿠키. 브라우저를 닫아도 **파일 형태로 저장**되어 시간이 지나기 전까지 유지됩니다. (예: "오늘 하루 보지 않기", "로그인 기억하기")
  - 쿠키의 보안 속성(HttpOnly, Secure, SameSite)은 각각 어떤 공격을 방어하나요? 🍠
    - **HttpOnly:** **XSS(Cross-Site Scripting) 방어.** 자바스크립트(`document.cookie`)로 쿠키를 읽는 것을 금지하여 해커가 세션 ID를 훔쳐가는 것을 막습니다.
    - **Secure:** **스니핑(Sniffing) 방어.** HTTPS 연결에서만 쿠키를 전송하게 하여, 네트워크 통신 중간에 정보를 가로채는 것을 막습니다.
    - **SameSite:** **CSRF(Cross-Site Request Forgery) 방어.** 다른 사이트에서 내가 로그인한 사이트로 요청을 보낼 때 쿠키가 자동으로 딸려가는 것을 막습니다.
  - 쿠키의 한계점(용량, 보안, 네트워크, 도메인 제약)을 정리해주세요. 🍠
    - **용량:** 한 개당 **4KB** 이내로 매우 작음.
    - **보안:** 클라이언트에 저장되므로 **사용자가 변조** 가능함. (매우 위험!)
    - **성능:** 모든 요청에 자동으로 포함되므로, 쿠키가 많아지면 네트워크 오버헤드가 커짐.
    - **도메인 제약:** 기본적으로 생성한 도메인 외에는 접근 불가 (Cross-domain 이슈).
  - 쿠키만으로 상태 관리를 해결할 수 없는 이유는 무엇인가요? 🍠
    **"데이터의 신뢰성"** 때문입니다.
    쿠키는 사용자가 브라우저 개발자 도구로 값을 마음대로 바꿀 수 있습니다.
    예를 들어 쿠키에 `is_admin=false` 라고 적혀있는데 사용자가 이를 `true` 로 바꾸면 서버가 속아 넘어갈 수 있습니다.
    따라서 **진짜 중요한 정보(사용자 권한, 개인정보 등)는 서버(세션)나 암호화된 토큰(JWT)에 저장**해야 합니다.

  [개발자 매튜 | XSS(Cross-Site Scripting) 크로스 사이트 스크립팅 공격이란?](https://www.yolog.co.kr/post/security-xss/)
  - XSS 공격은 무엇인가요?
    공격자가 웹 사이트에 **악성 자바스크립트 코드를 삽입**하여, 해당 사이트를 방문하는 다른 사용자의 브라우저에서 그 코드가 실행되게 만드는 공격입니다.
    - **공격의 본질:** 브라우저는 서버가 보내준 HTML에 포함된 스크립트가 '개발자가 짠 코드'인지 '해커가 삽입한 코드'인지 구분하지 못하고 무조건 실행한다는 점을 악용합니다.
    - **주요 피해:**
      - **쿠키 탈취:** `document.cookie`를 읽어 세션 ID를 해커 서버로 전송 (세션 하이재킹).
      - **피싱:** 가짜 로그인 폼을 띄워 비밀번호 수집.
      - **스크립트 실행:** 사용자 권한으로 게시글 삭제, 송금 요청 등 수행.
    ***
    ## XSS 공격의 3가지 유형
    1. **Stored XSS (저장형):** 악성 스크립트가 **서버 DB에 저장**되었다가, 페이지를 조회하는 모든 사용자에게 실행됨. (예: 게시판 댓글)
    2. **Reflected XSS (반사형):** URL 파라미터에 스크립트를 포함시켜, 서버가 이를 그대로 응답에 담아 보낼 때 실행됨. (예: 검색 결과창에 `검색어 <script>...` 표시)
    3. **DOM-based XSS:** 서버를 거치지 않고, 브라우저에서 자바스크립트가 URL의 해시(`#`) 등을 읽어 화면에 띄울 때 발생함.
    ***
    ##
  - XSS 방어 전략에 대해 정리해주세요.
    ### ① HTML 이스케이프 (Escape) - 가장 기본
    HTML 특수 문자를 브라우저가 코드로 인식하지 못하도록 **문자열로 치환**하는 방법입니다.
    - `<` → `&lt;` / `>` → `&gt;` / `&` → `&amp;` / `"` → `&quot;` / `'` → `&#39;`
    - **주의사항:** 블로그 내용처럼 `&`를 가장 먼저 치환해야 이중 치환 오류를 막을 수 있습니다.
    ### ② HTML 새니타이즈 (Sanitize) - 리치 에디터용
    블로그 글이나 댓글에 `<b>`, `<a>` 같은 태그는 허용하고 싶을 때 사용합니다.
    - **DOMPurify** 같은 라이브러리를 사용하여 `<script>`, `onerror` 같은 위험한 요소만 골라내서 제거(살균)합니다.
    ### ③ CSP (Content Security Policy) - 강력한 보조벽 (중요!)
    웹 페이지에서 실행할 수 있는 **스크립트의 출처를 제한**하는 보안 정책을 HTTP 헤더에 설정하는 것입니다.
    - 예: `Content-Security-Policy: script-src 'self' https://trusted.com;`
    - 이렇게 설정하면 해커가 외부에서 가져온 스크립트나 인라인 스크립트(`<script>alert(1)</script>`)는 브라우저가 실행을 거부합니다.
    ### ④ HttpOnly 쿠키 설정
    XSS 방어의 최후의 보루입니다. 자바스크립트가 쿠키에 접근하는 것을 막아, 설령 XSS 공격에 뚫리더라도 **세션 ID만큼은 탈취당하지 않게** 보호합니다.
  - 추가 정리 ⭐
    - **React/Vue의 자동 방어:** 현대적인 프레임워크는 기본적으로 변수를 출력할 때 **자동 이스케이프**를 해줍니다. 하지만 `dangerouslySetInnerHTML` 같은 속성을 쓸 때는 이 방어막이 해제되므로 주의해야 합니다.
    - **검증(Validation)과 출력(Sanitization)의 차이:** 입력 단계에서 막는 것도 중요하지만, **화면에 보여주기 직전(출력 단계)**에 이스케이프나 새니타이즈를 하는 것이 가장 안전합니다. 데이터는 어디서든 오염될 수 있기 때문입니다.

  [개발자 매튜 | 세션 하이재킹 완벽 가이드 - 쿠키 탈취(XSS) 공격과 HttpOnly 방어법](https://www.yolog.co.kr/post/security-session-hijacking/)
  - 세션 하이재킹은 무엇인가요? 🍠
    운항 중인 비행기를 납치하는 '하이재킹'처럼, **이미 인증되어 활성화된 사용자의 '세션'을 가로채는 공격**입니다.
    - **원리:** HTTP는 무상태(Stateless)라 서버는 **세션 ID**만 보고 사용자를 식별합니다. 공격자가 어떤 수단(주로 XSS)으로든 사용자의 세션 ID를 훔쳐내어 자신의 브라우저에 심으면, 아이디/비밀번호 없이도 **피해자의 계정으로 로그인한 상태**가 됩니다.
    - **비유:** 놀이공원에서 누군가 내 손목에 채워진 '자유이용권(세션 ID)'을 몰래 끊어서 자기 손목에 차고 나인 척 돌아다니는 것과 같습니다.
  - XSS 공격을 차단하는 방법은 무엇인가요? 🍠
    - **HTML 이스케이프:** 사용자가 입력한 특수문자(`<`, `>`, `&` 등)를 코드가 아닌 일반 문자로 변환합니다.
    - **HttpOnly 쿠키 설정:** **가장 중요한 방어선입니다.** 자바스크립트가 `document.cookie`로 쿠키를 읽는 것 자체를 막아버립니다. 공격자가 XSS에 성공해서 스크립트를 실행하더라도 세션 ID는 훔쳐갈 수 없게 됩니다.
    - **CSP(Content Security Policy) 도입:** 허용되지 않은 출처의 스크립트 실행을 브라우저 단에서 거부합니다.
  - 쿠키의 다양한 옵션들에 대해서 정리해주세요. 🍠
    쿠키를 생성할 때(`Set-Cookie`) 설정할 수 있는 옵션들입니다. 각 옵션이 어떤 공격을 막는지 연결해서 파악하는 것이 중요 !
    | **옵션**              | **의미**                           | **방어하는 공격**                         |
    | --------------------- | ---------------------------------- | ----------------------------------------- |
    | **HttpOnly**          | 자바스크립트 접근 금지             | **XSS**를 통한 쿠키 탈취 방지             |
    | **Secure**            | HTTPS 환경에서만 전송              | **네트워크 스니핑**(중간에 가로채기) 방지 |
    | **SameSite**          | 외부 사이트 요청 시 쿠키 전송 제한 | **CSRF**(사이트 간 요청 위조) 방지        |
    | **Max-Age / Expires** | 쿠키의 유효 기간 설정              | **세션 고정 공격** 및 장기 탈취 피해 방지 |
    | **Path / Domain**     | 쿠키가 전송될 경로와 도메인 제한   | 불필요한 정보 노출 방지                   |
  - 추가 정리 ⭐
    - **세션 고정 공격(Session Fixation) 방어:**
      - 공격자가 미리 만들어둔 세션 ID를 사용자에게 강제로 심어두고, 사용자가 그 ID로 로그인하면 공격자도 같이 로그인되는 공격입니다.
      - **방어법:** 사용자가 **로그인에 성공하는 즉시 새로운 세션 ID를 발급**해야 합니다. (블로그의 '세션 재생성' 내용)
    - **HTTPS만으로는 부족한 이유:**
      - HTTPS는 통신 구간을 암호화하여 '스니핑'은 막아주지만, 이미 브라우저에 들어온 악성 스크립트(XSS)가 쿠키를 읽어가는 것까지는 막지 못합니다. 그래서 **HTTPS(Secure)와 HttpOnly는 반드시 세트**로 움직여야 합니다.
    - **예측 불가능한 세션 ID:**
      - 단순히 `Date.now()`(시간값)를 쓰면 공격자가 다음 세션 ID를 추측하기 쉽습니다. 반드시 암호학적으로 안전한 **난수 생성기(Crypto)**를 써야 합니다.

  # 📚 블로그 읽고 CSRF 공격 정리해보기
  ***
  [개발자 매튜 | 프론트엔드·백엔드 개발자라면 꼭 알아야 할 CSRF 공격과 방어법](https://www.yolog.co.kr/post/security-csrf/)
  - CSRF는 무엇인가요? 🍠
    **"사이트 간 요청 위조"**라는 뜻으로, 사용자가 자신의 의지와 무관하게 공격자가 의도한 행위(송금, 비밀번호 변경 등)를 특정 웹사이트에 요청하게 만드는 공격입니다.
    - **핵심 원리:** 브라우저가 특정 사이트로 요청을 보낼 때, 그 사이트의 **쿠키를 자동으로 함께 전송**한다는 점을 악용합니다.
    - **성공 조건:** 1. 사용자가 대상 사이트에 **로그인**되어 있어 브라우저에 세션 쿠키가 있어야 함.
    2. 사용자가 공격자가 만든 **악성 페이지/링크**를 클릭해야 함.
    - **비유:** 내가 은행에 로그인한 상태에서 해커가 보낸 "무료 쿠폰" 버튼을 눌렀더니, 내 브라우저가 나 대신 은행에 "해커에게 1억 송금해줘"라는 편지를 쓰고, 내 인감도장(쿠키)까지 찍어서 보내버리는 상황입니다.
  - CSRF 방어 전략에 대해 정리해주세요. 🍠
    | **방어 전략**             | **동작 원리**                                                          | **특징**                                                            |
    | ------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------- |
    | **SameSite 쿠키**         | 쿠키 옵션에 `Strict`나 `Lax`를 설정해 타 사이트에서의 쿠키 전송을 제한 | 가장 쉽고 효과적이지만 구형 브라우저 지원 확인 필요                 |
    | **CSRF 토큰**             | 서버가 발행한 무작위 토큰을 폼(Form)에 심고, 요청 시 서버의 값과 대조  | **가장 강력하고 표준적인 방법.** 브라우저 지원 여부와 무관하게 작동 |
    | **Referer/Origin 확인**   | HTTP 헤더를 통해 요청이 온 주소가 정상적인 우리 사이트인지 검사        | 구현이 간단하지만 헤더 변조나 누락 가능성 있음                      |
    | **재인증 (reCAPTCHA 등)** | 민감한 작업 시 비밀번호 재입력이나 캡차를 요구                         | 보안성은 최상이지만 사용자 경험(UX)이 불편해짐                      |
  - CSRF 토큰의 장점과 단점에 대해 정리해주세요. 🍠
    - **장점 (강력함):**
      - 브라우저의 쿠키 정책에 의존하지 않고 서버가 직접 통제하므로 매우 안전합니다.
      - 공격자는 사용자의 브라우저 속에 숨겨진 이 무작위 토큰값을 읽어낼 수 없기 때문에 요청 위조가 불가능합니다.
    - **단점 (복잡함):**
      - 모든 `POST`, `PUT`, `DELETE` 요청마다 토큰을 폼이나 헤더에 실어 보내야 하므로 개발 공수가 많이 듭니다.
      - 서버 메모리(세션)에 토큰을 저장해야 하므로 서버 부하가 약간 늘어날 수 있습니다.
  - CAPTCHA는 무엇인가요? 🍠
    **"사람과 컴퓨터를 구별하기 위한 자동화된 테스트"**의 약자입니다.
    - **역할:** 봇(Bot)이나 자동화된 스크립트가 CSRF 공격을 대량으로 시도하거나 회원가입 테러를 하는 것을 막습니다.
    - **발전:** 과거에는 읽기 힘든 글자를 타이핑하게 했으나(v1), 최근에는 "로봇이 아닙니다" 체크박스(v2)나 사용자 모르게 점수를 매기는 방식(v3)으로 발전하여 보안과 편의성을 동시에 잡고 있습니다.
  - 추가 정리 ⭐
    - **GET 요청은 안전해야 한다:**
      - 블로그 글에도 슬쩍 언급되었지만, **GET 요청으로는 절대로 데이터를 변경(송금 등)하면 안 됩니다.** GET은 오직 조회용으로만 써야 CSRF 공격 범위를 크게 줄일 수 있습니다. (Lax 옵션이 GET은 허용하기 때문입니다!)
    - **Custom Header 검증:**
      - 최근 SPA(React, Vue 등) 환경에서는 요청 헤더에 `X-Requested-With` 같은 커스텀 헤더를 추가하는 것만으로도 상당수의 CSRF를 막을 수 있습니다. 브라우저의 **CORS 정책** 덕분에 다른 도메인의 스크립트는 이런 커스텀 헤더를 마음대로 붙일 수 없기 때문입니다.
    - **XSS와 CSRF의 관계:**
      - **XSS**는 우리 사이트 내부에서 스크립트를 실행하는 공격이고, **CSRF**는 밖에서 우리 사이트로 요청을 던지는 공격입니다. 만약 사이트에 **XSS 취약점이 있다면 CSRF 토큰마저 탈취**당할 수 있으므로, 항상 XSS 방어(HttpOnly 등)가 선행되어야 합니다.
  [[🔐인증] 3. 쿠키 vs 헤더: 데이터를 어떻게 전송할까](https://velog.io/@cucumtomatobean/%EC%9D%B8%EC%A6%9D-3.-%EC%BF%A0%ED%82%A4-vs-%ED%97%A4%EB%8D%94-%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%A5%BC-%EC%96%B4%EB%96%BB%EA%B2%8C-%EC%A0%84%EC%86%A1%ED%95%A0%EA%B9%8C)
  - 쿠키 방식으로 인증 정보를 전송할 때 브라우저가 자동으로 쿠키를 포함하는 조건은 무엇인가요? 🍠
    브라우저가 서버로 쿠키를 자동으로 실어 보내려면 다음 조건이 충족되어야 합니다.
    - **도메인 일치:** 요청을 보내는 곳과 쿠키의 `Domain` 설정이 일치해야 합니다. (기본적으로 서브도메인이 다르면 다른 도메인으로 간주합니다.)
    - **경로 일치:** 요청 URL이 쿠키의 `Path` 설정 범위 내에 있어야 합니다.
    - **유효 기간:** 쿠키가 만료(`Max-Age`, `Expires`)되지 않은 상태여야 합니다.
    - **보안 속성:** HTTPS 연결이 아닐 때 `Secure` 쿠키는 전송되지 않으며, `SameSite` 설정에 따라 타 사이트에서의 전송이 제한될 수 있습니다.
  - 크로스 도메인 환경에서 쿠키를 전송하려면 서버와 클라이언트 측에서 각각 어떤 설정이 필요할까요? 🍠
    도메인이 다른 두 환경(예: `app.a.com` → `api.a.com`)에서 쿠키를 주고받으려면 양쪽의 합의가 필요합니다.
    - **서버(Back-end):**
      - `Access-Control-Allow-Credentials: true` 헤더를 응답에 포함해야 합니다.
      - **주의:** 이때 `Access-Control-Allow-Origin`은 와일드카드()를 쓸 수 없고, 반드시 특정 도메인을 명시해야 합니다.
    - **클라이언트(Front-end):**
      - 요청 시 `credentials: 'include'`(fetch) 또는 `withCredentials: true`(axios) 옵션을 명시적으로 켜야 합니다.
  - 쿠키 기반 인증에서 CSRF 공격이 발생할 수 있는 원리는 무엇인가요? 🍠
    쿠키의 **'자동 전송'**이라는 편리함이 독이 되는 경우입니다.
    - **원리:** 브라우저는 요청이 어디서(어떤 탭에서) 시작되었는지 따지지 않고, 목적지가 `bank.com`이면 무조건 `bank.com`의 쿠키를 꺼내서 보냅니다. 공격자가 만든 가짜 사이트에서 `bank.com`으로 요청을 보내게 유도하면, 브라우저는 사용자의 인감도장(로그인 쿠키)을 찍어서 보내버리는 것입니다.
  - 헤더 방식 인증의 주요 장점(예: CSRF 방어, 선택적 전송, CORS 단순화)을 정리해주세요. 🍠
    개발자가 직접 `Authorization` 헤더에 토큰을 넣는 방식의 장점입니다.
    - **CSRF 방어:** 공격자가 요청을 보낼 수는 있어도, 자바스크립트로 내 토큰값을 직접 꺼내서 헤더에 넣어줄 수는 없으므로(동일 출처 정책 때문) 공격이 성립하지 않습니다.
    - **선택적 전송:** 이미 지나가는 이미지나 정적 파일 요청에는 토큰을 안 보내고, 필요한 API에만 골라서 보낼 수 있어 효율적입니다.
    - **CORS 단순화:** 쿠키를 안 쓰면 `allowCredentials`를 신경 쓰지 않아도 되므로 서버 설정이 훨씬 자유롭습니다.
  - 토큰을 클라이언트에 저장할 때 LocalStorage, SessionStorage, 메모리 저장의 장단점을 비교해주세요. 🍠
    | **저장소**         | **장점**                               | **단점**                                  | **보안** |
    | ------------------ | -------------------------------------- | ----------------------------------------- | -------- |
    | **LocalStorage**   | 구현이 매우 쉽고 새로고침해도 유지됨   | **XSS 공격**에 매우 취약 (JS로 읽기 가능) | 위험 🚨  |
    | **SessionStorage** | 탭별로 관리되어 약간 더 안전함         | 탭을 닫으면 로그아웃됨, 여전히 XSS에 취약 | 보통 ⚠️  |
    | **메모리(변수)**   | **가장 안전함.** JS 외부에서 접근 불가 | 새로고침하면 로그아웃됨                   | 안전 ✅  |
  - Single Page Application, Mobile Application, Server Side Rendering Application 에서 여러분들이 생각하는 적합한 인증 전략은 무엇이라고 생각하시나요? 🍠
    - **SPA (CSR):** **메모리 저장 + Refresh Token(HttpOnly 쿠키)** 조합을 추천합니다. Access Token은 메모리에 두어 XSS를 막고, 새로고침 시 사라지는 문제는 안전한 쿠키에 담긴 Refresh Token으로 서버에서 조용히 새로 받아오는 방식입니다.
    - **Mobile App:** **시스템 보안 저장소(Keychain/Keystore)**가 정답입니다. 브라우저가 아니므로 쿠키 개념이 약하며, OS가 제공하는 금고에 토큰을 넣고 헤더 방식으로 통신하는 것이 표준입니다.
    - **SSR (Next.js 등):** **HttpOnly 쿠키** 방식이 가장 유리합니다. 첫 화면을 그릴 때 서버가 쿠키를 바로 읽어서 로그인된 화면을 그려줄 수 있기 때문입니다. 헤더 방식은 클라이언트 자바스크립트가 실행될 때까지 기다려야 하는 단점이 있습니다.
  - 추가 정리 ⭐
    - **Bearer는 무슨 뜻인가요?** `Authorization: Bearer <token>`에서 Bearer는 "이 토큰을 소지한 사람(Bearer)에게 권한을 부여하라"는 뜻의 약속된 문구입니다.
    - **Preflight 요청:** CORS 환경에서 헤더에 토큰을 담아 보내면, 브라우저는 실제 요청 전에 `OPTIONS` 메서드로 **Preflight** 요청을 먼저 보냅니다. 서버는 이 `OPTIONS` 요청에도 적절한 응답을 줘야 통신이 성공합니다.

            **1. 이메일 로그인 (전통적인 방식)**

                3. 일치하면 서버는 **세션 ID**를 생성하여 쿠키에 담아주거나, **JWT**를 생성하여 전달합니다.

                ◦ **비밀번호 책임:** 서비스 운영자가 비밀번호 유출 방지(해싱, 솔팅 등)의 모든 책임을 집니다.

            **2. 카카오로 계속하기 (OAuth 2.0 방식)**
            이 방식은 인증 업무를 믿을 수 있는 제3자(카카오)에게 **위임**하는 방식입니다.
            • **동작 방식 (OAuth 2.0 프로토콜):**
                1. 사용자가 '카카오 로그인' 버튼을 누르면 **카카오 인증 서버**로 이동합니다.
                2. 사용자는 카카오에 로그인을 하고, `matthew.com`이 내 프로필 정보를 가져가는 것에 동의합니다.

                4. `matthew.com`은 이 코드를 카카오에 제출하고, 대신 **액세스 토큰(Access Token)**을 받습니다.
                5. 이 토큰을 사용해 카카오로부터 사용자 정보(이름, 이메일 등)를 받아와서 로그인을 처리합니다.
            • **특징:**
                ◦ **비밀번호 불필요:** `matthew.com`은 사용자의 카카오 비밀번호를 절대 알 수 없습니다. 토큰만 주고받을 뿐입니다.
                ◦ **사용자 편의성:** 여러 사이트의 비밀번호를 일일이 기억할 필요가 없습니다.
            • **보안 이점:** 카카오라는 거대 플랫폼의 강력한 보안(2차 인증 등)을 그대로 활용할 수 있습니다.

            | **구분** | **이메일 로그인 (ID/PW)** | **카카오 로그인 (OAuth)** |
            | --- | --- | --- |
            | **인증 주체** | 자체 서비스 서버 | 카카오 (인증 대행) |
            | **핵심 데이터** | 사용자 아이디 + **비밀번호** | 카카오에서 발급한 **액세스 토큰** |
            | **비밀번호 관리** | 서비스 DB에 직접 저장/관리 | 관리 안 함 (알 수도 없음) |
            | **로그인 유지** | 자체 발급한 세션/JWT | 카카오 정보를 바탕으로 발급한 세션/JWT |

            ## 2. 보안적 장치와 법적 요구사항의 배치
            
            우리가 배운 보안 기술들을 적용하기 위해 반드시 들어가야 하는 요소들이 UI의 위치를 결정합니다.

            - **약관 및 개인정보 동의:** 특히 소셜 로그인 버튼 하단에는 "로그인 시 이용약관에 동의하는 것으로 간주합니다"라는 문구가 필수적으로 들어갑니다.
            
            ## 3. 소셜 로그인(OAuth)의 정책적 가이드라인
            
            "카카오로 로그인", "Google로 로그인" 버튼이 어느 사이트나 똑같이 생긴 이유는 해당 **플랫폼(카카오, 구글 등)에서 강제하는 디자인 가이드라인** 때문입니다.
            
            - **브랜드 아이덴티티 유지:** 카카오는 노란색, 구글은 흰색 바탕에 로고 등 고유의 색상과 로고 비율을 지키지 않으면 API 사용 승인을 해주지 않는 경우도 있습니다.
            - **신뢰감 형성:** 사용자는 자신이 평소 쓰는 서비스의 공식 로고를 확인해야 "아, 내 정보를 안전하게 카카오가 인증해주겠구나"라고 안심하고 클릭(Trust)하게 됩니다.

            - **자동 완성 방지:** 보안이 중요한 사이트의 입력창에는 `autocomplete="off"` 같은 속성이 숨어 있어, 브라우저가 비밀번호를 자동으로 채우는 것을 제어하기도 합니다.
